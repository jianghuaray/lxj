# 项目代码与性能审查问题汇总

审查时间：2026-05-01  
审查范围：`frontend`、`backend` 主要业务代码、构建产物与接口实现  
审查方式：源码静态审查 + 前端生产构建验证

## 总结

本项目当前的主要问题集中在 4 类：

1. 数据持久化与一致性风险
2. 前后端接口字段契约不一致
3. 统计报表数据真实性不足
4. 后端查询与前端打包性能存在明显优化空间

其中优先级最高的问题有：

- 系统配置与操作日志只保存在内存中，服务重启后会全部丢失
- 工单号生成逻辑存在并发冲突风险
- 看板部分统计数据由前端随机生成，报表结果不可信

---

## 详细问题

### 1. [P0] 系统配置与操作日志只存在内存里，重启即丢失

- 文件位置：`backend/src/routes/settings.js:6-28`
- 问题描述：
  `serviceTypes`、`serviceAreas`、`sourceChannels`、`cancelReasons` 和 `operationLogs` 都是模块级变量，没有持久化到数据库或文件存储。
- 影响：
  - 管理员在系统设置页做的修改，在服务重启后会全部丢失
  - 操作日志无法长期保留
  - 生产环境下容易出现“页面改过但重启后恢复默认”的问题
- 风险等级：高
- 建议：
  - 增加 `settings` 与 `operation_logs` 表，统一落库
  - 如果暂时不建表，至少落盘到可恢复的持久化文件中

### 2. [P1] 客户接口返回字段与前端约定不一致，页面会长期显示空值或 0

- 文件位置：`backend/src/routes/customers.js:26-38`
- 问题描述：
  客户列表直接返回 Sequelize 实体，详情接口也是 `...c` 加少量 camelCase 字段；但前端读取的是 `totalOrders`、`totalAmount`、`lastOrderAt`、`sourceChannel`、`createdAt` 等 camelCase 字段。
- 影响：
  - 客户列表中的累计工单、累计消费、最近报修等字段可能显示为空
  - 客户详情中的首次来源、注册时间等字段可能不正确
  - 页面看起来“能打开”，但展示数据不可信
- 风险等级：高
- 建议：
  - 后端统一做字段映射，输出稳定的 camelCase DTO
  - 不要让前端直接依赖 Sequelize 原始字段结构

### 3. [P1] 客户详情页继续读取不存在的订单字段，历史记录金额和时间会错

- 文件位置：`frontend/src/views/CustomerDetail.vue:113-119`
- 问题描述：
  后端详情接口给每条订单返回的是 `totalFee` 和 `createdAt`，但前端渲染时读取的是 `order.amount` 和 `order.receivedAt`。
- 影响：
  - 工单历史中的费用会显示为 `0`
  - 工单时间会显示为空或错误
- 风险等级：高
- 建议：
  - 前端改为读取 `totalFee`、`createdAt`
  - 同时与客户接口字段映射一并统一

### 4. [P1] 看板区域分析在前端造随机数，报表每次刷新都不一致

- 文件位置：`frontend/src/views/Dashboard.vue:293-305`
- 问题描述：
  `fetchAreaStats()` 没有请求真实统计接口，而是直接基于区域工单数叠加 `Math.random()` 生成完成率、满意度和营收。
- 影响：
  - 同一时间段刷新页面会得到不同结果
  - 报表数据不具备业务参考价值
  - 会显著降低用户对数据看板的信任
- 风险等级：高
- 建议：
  - 由后端提供真实区域统计接口
  - 前端只负责展示，不要生成业务统计值

### 5. [P1] 工单号用“当日 count + 1”生成，并发创建时会撞唯一键

- 文件位置：`backend/src/routes/workOrders.js:282-295`
- 问题描述：
  当前逻辑先统计当天工单数量，再拼接生成新工单号。如果两个请求同时进入，可能读到相同计数，最终生成相同 `order_no`。
- 影响：
  - 并发下单时会间歇性创建失败
  - 难以复现但线上一定会出现
  - 用户会看到偶发性的“新增工单失败”
- 风险等级：高
- 建议：
  - 使用数据库保证唯一性的序列方案
  - 或增加独立流水号表/事务锁
  - 或使用时间戳 + 随机段/雪花 ID 等更稳妥策略

### 6. [P1] 师傅列表和排行存在明显 N+1 查询，数据量上来后会变慢

- 文件位置：`backend/src/routes/technicians.js:39-70`
- 问题描述：
  师傅列表接口先查一页师傅，再对每个师傅分别执行一次工单数统计和一次满意度聚合；排行接口也重复执行类似逻辑。
- 影响：
  - 查询次数随师傅数量线性增长
  - 数据量稍大后响应时间会明显上升
  - 首页和列表页性能会逐渐变差
- 风险等级：高
- 建议：
  - 改为按 `technician_id` 分组聚合，一次性取回统计结果
  - 避免在循环中执行数据库查询

### 7. [P2] 前端包体过大，Dashboard 单页已超过 1MB

- 文件位置：`frontend/src/views/Dashboard.vue:200-202`
- 相关位置：
  - `frontend/src/main.js`
  - `frontend/src/styles/index.scss`
- 问题描述：
  `Dashboard.vue` 直接 `import * as echarts from 'echarts'`，入口又全量注册 `@element-plus/icons-vue`，同时引入整包 Element Plus 样式，导致打包体积偏大。
- 构建验证结果：
  - `Dashboard` 路由产物约 `1.05 MB`
  - 主包约 `1.21 MB`
  - 已触发 Vite chunk size warning
- 影响：
  - 首次加载慢
  - 看板页切换耗时更长
  - 移动端和较弱设备体验更差
- 风险等级：中
- 建议：
  - ECharts 按需引入
  - 避免全量注册 Element Plus 图标
  - 结合 `manualChunks` 或更细粒度懒加载进行拆包

## 额外性能观察

### 工单列表接口偏重

- 文件位置：`backend/src/routes/workOrders.js:17-29, 58-63`
- 问题：
  - 先查 `Construction.findAll` 再组装 `IN` 条件过滤工单
  - 每次请求还会循环执行 7 次 `WorkOrder.count`
- 影响：
  在筛选条件变复杂或数据量变大时，列表接口会变慢。

### 构建存在 Sass 弃用告警

- 文件位置：`frontend/src/styles/index.scss:1-3`
- 问题：
  当前仍在使用 Sass `@import` 和 legacy JS API。
- 影响：
  当前不阻塞构建，但后续升级 Sass 时会增加维护成本。

---

## 建议修复优先级

建议按下面顺序处理：

1. 配置持久化与操作日志落库
2. 客户模块前后端字段契约统一
3. 看板区域统计改为真实后端数据
4. 工单号生成逻辑改为并发安全方案
5. 师傅列表与排行接口聚合优化，消除 N+1 查询
6. 前端按需加载与拆包优化
