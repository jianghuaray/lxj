## 二、P0 紧急问题

### 2.1 后端安全漏洞

#### 问题 1：CORS 完全开放

- **文件**：`backend/src/app.js` 第 20 行
- **现状**：`app.use(cors())` 允许任何域名访问 API
- **风险**：CSRF 攻击、数据泄露
- **修复方案**：

```javascript
app.use(cors({
  origin: process.env.FRONTEND_URL || 'http://localhost:5173',
  credentials: true
}));
```

#### 问题 2：登录接口无速率限制

- **文件**：`backend/src/routes/auth.js` 第 10-51 行
- **现状**：`POST /login` 无任何速率限制，可无限暴力破解
- **修复方案**：

```javascript
const rateLimit = require('express-rate-limit');
const loginLimiter = rateLimit({
  windowMs: 15 * 60 * 1000,
  max: 5,
  message: { error: '登录尝试过多，请稍后再试' }
});
router.post('/login', loginLimiter, async (req, res) => { ... });
```

#### 问题 3：JWT_SECRET 未做存在性校验

- **文件**：`backend/src/middleware/auth.js` 第 12 行、`backend/src/routes/auth.js` 第 34 行
- **现状**：`process.env.JWT_SECRET` 未设置时使用 `undefined` 作为密钥
- **修复方案**：在应用启动时校验

```javascript
if (!process.env.JWT_SECRET) {
  console.error('FATAL: JWT_SECRET is not set');
  process.exit(1);
}
```

#### 问题 4：请求体无大小限制

- **文件**：`backend/src/app.js` 第 21 行
- **现状**：`express.json()` 未设置 body 大小限制
- **风险**：攻击者可发送超大请求体耗尽内存
- **修复方案**：

```javascript
app.use(express.json({ limit: '1mb' }));
```

### 2.2 数据一致性：关键操作缺少事务

#### 问题 5：派单操作未使用事务

- **文件**：`backend/src/routes/workOrders.js` 第 318-354 行
- **现状**：先更新工单状态为 `dispatched`，再创建 Construction 记录。如果第二步失败，工单状态已变但没有施工记录
- **修复方案**：

```javascript
const transaction = await sequelize.transaction();
try {
  order.status = 'dispatched';
  await order.save({ transaction });
  await Construction.create({ ... }, { transaction });
  await transaction.commit();
} catch (error) {
  await transaction.rollback();
  throw error;
}
```

#### 问题 6：回访提交未使用事务

- **文件**：`backend/src/routes/workOrders.js` 第 581-622 行、`backend/src/routes/callbacks.js` 第 153-235 行
- **现状**：创建回访记录和更新工单状态不在同一事务中
- **修复方案**：同上，使用事务包裹

#### 问题 7：创建工单时客户创建和工单创建未使用事务

- **文件**：`backend/src/routes/workOrders.js` 第 257-315 行
- **现状**：如果客户不存在则先创建客户，再创建工单。工单创建失败会留下孤立客户记录
- **修复方案**：同上，使用事务包裹

### 2.3 内存/性能：全表加载到应用层

#### 问题 8：区域统计接口全表加载到内存

- **文件**：`backend/src/routes/dashboard.js` 第 176-252 行
- **现状**：先查出所有 Construction 记录，再查出所有 WorkOrder 的 id 和 area，再查出所有 CallbackRecord，然后在 JavaScript 中做关联和聚合
- **风险**：数据量大时导致严重性能问题和内存溢出
- **修复方案**：使用一条 SQL JOIN + GROUP BY 查询

```sql
SELECT w.area,
       COUNT(*) as total,
       SUM(CASE WHEN w.status IN ('completed','callback') THEN 1 ELSE 0 END) as completed,
       AVG(cr.satisfaction_score) as avgSatisfaction,
       SUM(c.service_fee) as revenue
FROM work_orders w
LEFT JOIN constructions c ON c.order_id = w.id
LEFT JOIN callback_records cr ON cr.order_id = w.id
WHERE w.area IS NOT NULL AND w.area != ''
GROUP BY w.area
```

#### 问题 9：师傅排行接口全量加载

- **文件**：`backend/src/routes/dashboard.js` 第 285-356 行
- **现状**：查出所有启用师傅（含全部施工记录），再批量查 Construction、CallbackRecord，共 3 次查询 + 大量 JS 计算
- **修复方案**：

```sql
SELECT t.id, t.name, t.specialties,
       COUNT(c.id) as orderCount,
       AVG(cr.satisfaction_score) as avgSatisfaction
FROM technicians t
LEFT JOIN constructions c ON c.technician_id = t.id
LEFT JOIN callback_records cr ON cr.order_id = c.order_id
WHERE t.status = 1
GROUP BY t.id
ORDER BY orderCount DESC
LIMIT 5
```

## 三、P1 高优先级问题

### 3.1 数据库缺少索引

以下高频查询字段未定义索引，随着数据增长查询性能将显著下降：

| 模型 | 缺少索引的字段 | 影响的查询 |
|------|---------------|-----------|
| **WorkOrder** | `status`, `customer_id`, `receiver_id`, `area`, `problem_category`, `completed_at`, `created_at` | 几乎所有列表查询、筛选、分组 |
| **Construction** | `order_id`, `technician_id` | 工单详情关联、师傅结算查询 |
| **CallbackRecord** | `order_id`, `callback_by` | 回访关联、满意度统计 |
| **OperationLog** | `operator_id`, `action`, `created_at` | 操作日志查询 |
| **Settings** | `category` | 每次设置查询 |

**修复示例（WorkOrder）**：

```javascript
indexes: [
  { fields: ['status'] },
  { fields: ['customer_id'] },
  { fields: ['receiver_id'] },
  { fields: ['area'] },
  { fields: ['problem_category'] },
  { fields: ['completed_at'] },
  { fields: ['created_at'] },
  { fields: ['status', 'created_at'] }  // 复合索引，用于超期查询
]
```

### 3.2 逻辑错误

#### 问题 13：待回访列表未排除已有回访记录的工单

- **文件**：`backend/src/routes/callbacks.js` 第 15-27 行
- **现状**：查询所有 `status='completed'` 的工单作为"待回访"，但没有排除已有 CallbackRecord 的工单
- **修复方案**：

```javascript
const completedOrders = await WorkOrder.findAll({
  where: {
    status: 'completed',
    '$callback.id$': null
  },
  include: [
    { model: Customer, as: 'customer' },
    { model: CallbackRecord, as: 'callback', required: false }
  ]
});
```

#### 问题 14：工单详情 callbackByName 取不到值

- **文件**：`backend/src/routes/workOrders.js` 第 231 行
- **现状**：`o.callback.callbackUser?.real_name` — 但 CallbackRecord 的 include 中没有 include User 模型，`callbackUser` 永远是 undefined
- **修复方案**：

```javascript
{
  model: CallbackRecord,
  as: 'callback',
  include: [{ model: User, as: 'callbackUser' }]
}
```

#### 问题 15：工单列表 include Customer 但未使用

- **文件**：`backend/src/routes/workOrders.js` 第 76 行
- **现状**：include 了 Customer 模型，但返回数据使用 WorkOrder 自身的 `customer_name` 和 `customer_phone`
- **修复方案**：移除不必要的 include，减少 JOIN 开销

#### 问题 16：authAdmin 未检查 req.user 是否存在

- **文件**：`backend/src/middleware/auth.js` 第 26-31 行
- **现状**：直接访问 `req.user.role`，如果 auth 中间件未被正确调用，`req.user` 为 undefined 导致 TypeError
- **修复方案**：

```javascript
const authAdmin = async (req, res, next) => {
  if (!req.user || req.user.role !== 'admin') {
    return res.status(403).json({ error: '需要管理员权限' });
  }
  next();
};
```

#### 问题 17：auth 中间件每次请求查数据库

- **文件**：`backend/src/middleware/auth.js` 第 13 行
- **现状**：`User.findByPk(decoded.userId)` 每次请求都查数据库验证用户是否存在
- **修复方案**：JWT token 中包含用户状态，或短期缓存用户信息
