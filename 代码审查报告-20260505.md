## 二、后端问题清单

### 🟡 P2 - 中等优先级

#### 问题 B-2.1：关键操作缺少事务处理（数据一致性风险）

- **文件**：
  - `backend/src/routes/workOrders.js`（派单、费用录入、回访）
  - `backend/src/routes/callbacks.js`（回访提交）
  - `backend/src/routes/constructions.js`
- **描述**：以下操作是多步变更但未包裹数据库事务：
  - 派单：修改工单状态 + 创建施工记录
  - 回访提交：创建回访记录 + 更新工单状态
  - 创建工单：创建客户（可选）+ 创建工单
- **风险**：第二步失败时，第一步的变更已提交，数据不一致
- **修复方案**：
  ```javascript
  const transaction = await sequelize.transaction()
  try {
    order.status = 'dispatched'
    await order.save({ transaction })
    await Construction.create({ ... }, { transaction })
    await transaction.commit()
  } catch (error) {
    await transaction.rollback()
    throw error
  }
  ```

#### 问题 B-2.2：dashboard 区域统计全表加载到内存（性能问题）

- **文件**：`backend/src/routes/dashboard.js`
- **描述**：先查出所有 Construction、所有 WorkOrder、所有 CallbackRecord，然后在 JS 中做关联和聚合
- **风险**：数据量大时 OOM（内存溢出）、响应缓慢
- **修复方案**：使用单条 SQL JOIN + GROUP BY 聚合

#### 问题 B-2.3：dashboard 师傅排行全量加载

- **文件**：`backend/src/routes/dashboard.js`
- **描述**：查出所有启用师傅及其全部施工记录，然后批量查询聚合
- **修复方案**：使用单条 SQL 聚合查询（LEFT JOIN + GROUP BY）

#### 问题 B-2.4：工单列表每次都查询统计数据（性能问题）

- **文件**：`backend/src/routes/workOrders.js`
- **描述**：每次翻页请求都重新计算 statusCounts, overdueCount, awaitCallbackCount
- **修复方案**：
  - 方案 A：将统计数据缓存（30秒 TTL）
  - 方案 B：移到独立的 `/orders/stats` 端点，前端按需请求
  - 方案 C：通过查询参数 `?includeStats=true` 控制是否返回统计

#### 问题 B-2.5：大量重复路由逻辑（代码可维护性）

- **文件**：
  - `workOrders.js`：POST /:id/assign vs /:id/dispatch
  - `workOrders.js`：PATCH vs PUT /:id/status
  - `workOrders.js`：POST fee vs PUT fees
  - `callbacks.js`：POST /:id/complete vs /:orderId
  - `technicians.js`：PUT vs PATCH
  - `customers.js`：手机号查重逻辑三处
- **修复方案**：提取公共处理函数

#### 问题 B-2.6：LIKE 查询中的通配符未转义

- **文件**：customers.js, workOrders.js, technicians.js, users.js, settings.js
- **描述**：用户可输入 `%` 或 `_` 操纵查询结果（虽无 SQL 注入但行为异常）
- **修复方案**：
  ```javascript
  function escapeLike(str) {
    return str.replace(/[%_]/g, '\\$&')
  }
  // 使用
  { name: { [Op.like]: `%${escapeLike(keyword)}%` } }
  ```

#### 问题 B-2.7：修改密码无强度检查

- **文件**：`backend/src/routes/auth.js`
- **描述**：POST /change-password 接受任意新密码，包括空字符串

#### 问题 B-2.8：重置密码默认值过于简单

- **文件**：`backend/src/routes/users.js`
- **描述**：未提供新密码时默认 `123456`

#### 问题 B-2.9：工单按师傅筛选时全量查 Construction

- **文件**：`workOrders.js`
- **描述**：先查出该师傅的所有 Construction，再筛选出 order_id
- **修复方案**：使用子查询或 include + where 替代

## 三、前端问题清单

### 🔴 P0 - 紧急问题

#### 问题 F-0.1：api.js 的 showError 函数无限递归（崩溃风险）

- **文件**：`frontend/src/utils/api.js` 第 16-22 行
- **代码**：
  ```javascript
  function showError(msg) {
    const now = Date.now()
    if (msg === lastErrorMsg && now - lastErrorTime < ERROR_DEDUPE_MS) return
    lastErrorMsg = msg
    lastErrorTime = now
    showError(msg)  // ← 这里！无限递归！
  }
  ```
- **风险**：无限递归导致调用栈溢出，页面崩溃
- **修复方案**：
  ```javascript
  function showError(msg) {
    const now = Date.now()
    if (msg === lastErrorMsg && now - lastErrorTime < ERROR_DEDUPE_MS) return
    lastErrorMsg = msg
    lastErrorTime = now
    // ← 这里调用 ElMessage
    ElMessage.error(msg)
  }
  ```

#### 问题 F-0.2：导出 Excel 时 pageSize: 9999 全量拉取

- **文件**：`frontend/src/views/orders/OrderList.vue` 第 411-454 行
- **描述**：
  - 导出时一次性请求所有数据
  - 无分页/分批逻辑
- **风险**：数据量大时：
  - 请求超时
  - 浏览器内存溢出/卡顿
  - 后端压力过大
- **修复方案**：
  - 方案 A（推荐）：后端提供流式导出端点
  - 方案 B：前端分批请求，每次 500 条，拼接后导出
  - 方案 C：限制最大导出数量，超过时提示用户缩小筛选范围

  ```javascript
      const response = await api.get('/orders', { params, signal })
      // ...
    } catch (error) {
      if (axios.isCancel(error)) return
      // ...
    }
  }
  ```

#### 问题 F-1.2：模板中直接调用函数导致不必要重渲染

- **文件**：`OrderList.vue` 第 125-127 行
- **代码**：
  ```html
  <span class="status-badge" :class="getStatusClass(order.status)">
    {{ getStatusText(order.status) }}
  </span>
  ```
- **问题**：每次渲染都调用函数，无法被 Vue 缓存
- **修复方案**：
  方案 1：改为 computed（推荐）
  方案 2：使用 v-once（对于订单列表这种基本不变的）
  方案 3：在数据获取时预先计算好这些值并存入对象

#### 问题 F-1.3：console.log 暴露敏感信息

- **文件**：多个组件
- **位置**：
  - `OrderList.vue` 导出错误：`console.error(error)`
  - `TechnicianAdd.vue`：可能有
- **修复方案**：
  - 移除或用条件编译（生产环境禁用）
  - 错误提示用 ElMessage 替代

#### 问题 F-1.4：401 处理状态同步

- **文件**：`frontend/src/utils/api.js` 第 56-68 行
