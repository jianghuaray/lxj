## 一、严重问题（P0 — 必须立即修复）

### 1. 🔴 前端 API 错误处理无限递归 — 必定崩溃

**文件：** `frontend/src/utils/api.js` 第 16-22 行

```javascript
function showError(msg) {
  const now = Date.now()
  if (msg === lastErrorMsg && now - lastErrorTime < ERROR_DEDUPE_MS) return
  lastErrorMsg = msg
  lastErrorTime = now
  showError(msg)  // ← 调用自己！应该调用 ElMessage.error(msg)
}
```

**问题：** `showError` 函数内部调用了 `showError(msg)` 而非 `ElMessage.error(msg)`，导致无限递归，任何 API 错误都会导致浏览器调用栈溢出崩溃。

**修复：** 将 `showError(msg)` 改为 `ElMessage.error(msg)`

**影响：** 所有 API 请求失败场景，用户看不到任何错误提示，页面直接白屏崩溃。

### 2. 🔴 删除操作缺少级联保护 — 数据不一致

**文件：** `backend/src/routes/customers.js` 第 293-306 行

```javascript
router.delete('/:id', auth, authAdmin, async (req, res) => {
  const customer = await Customer.findByPk(req.params.id);
  await customer.destroy(); // ← RESTRICT 约束会报错，但没处理
```

- 前端没有对应的业务提示，用户不知道为什么删除失败

**修复：** 在删除前检查是否有关联数据，或捕获 SequelizeForeignKeyConstraintError 返回友好提示

### 3. 🔴 工单号生成存在并发冲突风险

**文件：** `backend/src/routes/workOrders.js` 第 355-359 行

```javascript
const randomPart = String(Math.floor(Math.random() * 1000000)).padStart(6, '0');
const orderNo = `${datePart}${timePart}${randomPart}`;
```

**问题：** 
- 虽然使用了 6 位随机数，但 `Math.random()` 不是加密安全的
- 同一秒内并发创建工单时，随机碰撞概率约 1/1000000，对于小型系统可接受但不够健壮
- `order_no` 字段有 `unique` 约束，碰撞会导致 500 错误

**修复：** 使用数据库自增序列或 UUID，或在碰撞时自动重试

## 二、重要问题（P1 — 应尽快修复）

### 4. 🟠 师傅列表查询性能严重低下 — 多次串行查询

**文件：** `backend/src/routes/technicians.js` 第 10-250 行

**问题：** 获取师傅列表时，为了计算统计信息（工单数、满意度、营收），执行了 **4-6 次额外查询**：
1. `Technician.findAll()` — 获取师傅列表
2. `Construction.findAll()` — 工单数量统计
3. `Construction.findAll()` — 营收统计
4. `Construction.findAll()` — 获取 technician_id + order_id 映射
5. `CallbackRecord.findAll()` — 满意度统计

当排序字段为 `orderCount/avgSatisfaction/totalRevenue` 时，会先 **全量加载所有师傅** 再排序分页，完全绕过了数据库分页。

**修复建议：** 使用一条 JOIN + GROUP BY 查询替代多次查询：
```sql
SELECT t.*, COUNT(c.id) as orderCount, 
       AVG(cr.satisfaction_score) as avgSatisfaction,
       SUM(c.total_fee) as totalRevenue
FROM technicians t
LEFT JOIN constructions c ON ...
LEFT JOIN callback_records cr ON ...
GROUP BY t.id
```

### 7. 🟠 志愿者搜索缺少 SQL 注入防护

**文件：** `backend/src/routes/volunteers.js` 第 46-49 行

```javascript
if (keyword) {
  where[Op.or] = [
    { name: { [Op.like]: `%${keyword}%` } },   // ← 未转义
    { phone: { [Op.like]: `%${keyword}%` } }    // ← 未转义
  ];
}
```

**问题：** `keyword` 直接拼入 LIKE 模式，没有使用 `escapeLike()` 转义。用户输入 `%` 或 `_` 会改变 LIKE 语义。虽然 Sequelize 参数化了查询不会导致 SQL 注入，但 LIKE 通配符注入仍可能泄露数据。

**修复：** 使用已有的 `escapeLike(keyword)` 函数
